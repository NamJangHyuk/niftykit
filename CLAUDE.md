> Claude Code 세션 시작 시 이 파일을 참조하거나 프로젝트 루트의 CLAUDE.md에 포함시켜 사용하세요.

## 🎯 프로젝트 핵심 목표

> **"간단한 기능을 제대로 전달하는 도구를 만들고, 검색을 통해 유입된 사용자로부터 구글 애드센스 광고 수익을 얻는다."**

이 목표에 따라 모든 개발 우선순위는 다음 순서를 따릅니다.

1. **기능 신뢰성** — 도구가 딱 한 가지 일을 정확하고 빠르게 해낼 것 (기능이 많은 것보다 확실한 것이 우선)
2. **검색 상위 노출(SEO)** — 사용자가 이 도구를 실제로 검색해서 찾아올 수 있어야 수익이 발생 (아래 SEO 섹션이 사실상 이 프로젝트의 1순위 기술 과제)
3. **광고 수익화** — 검색 유입 사용자가 이탈하지 않고 광고를 자연스럽게 노출받도록 UX 설계
4. **서버 비용 0원** — GitHub Pages 기반 정적 사이트 유지

> ⚠️ Claude Code에 새 도구 개발을 요청할 때, "기능만 만들어 줘"라고 하지 말고 **"검색 상위 노출까지 고려해서"**를 항상 포함하세요. SEO는 나중에 추가하는 게 아니라 처음부터 설계에 포함되어야 효과가 있습니다.

## 1. 프로젝트 구조 (모듈형)

여러 도구를 하나의 리포지토리에서 관리합니다.

```text
/your-repo-name
  ├── index.html           (통합 대시보드: 모든 도구 링크)
  ├── privacy.html         (애드센스 승인 필수: 개인정보 처리방침)
  ├── /vector-tool
  ├── /stair-calc
  ├── /flag-master
  ├── /pt-timer
  ├── /tempo-counter
  ├── /emoji-search
  ├── /special-char
  ├── /barcode-gen
  └── ... (도구별 폴더, 각각 index.html / style.css / script.js 3분리 구조)
```

**규칙**
- 모든 도구는 `index.html / style.css / script.js` 3분리 구조 유지
- 순수 자바스크립트(서버 로직 없음), 필요 시 CDN(Unpkg/JSDelivr)만 사용
- 모바일 반응형 필수
- 각 도구는 **디자인·UX 상 완전히 독립된 하나의 앱**처럼 느껴지게 구성 (아래 2번 참조)

## 2. 앱별 독립 디자인 & UX 원칙

각 도구는 통합 대시보드에 속해 있지만, **사용자가 그 도구만 쓸 때는 그 도구만의 독립적인 서비스처럼 느껴지도록** 디자인합니다.

### 2-1. 공통 원칙
- 도구마다 목적에 맞는 **고유 컬러 테마/톤**을 부여 (예: 계산기류=차분한 블루/그레이 계열, 운동류=에너지틱한 오렌지/레드 계열, 시공/공학 도구=전문적인 다크모드 계열)
- 공통 대시보드에서 쓰는 공용 CSS 변수(색상, 폰트, 여백 단위)는 재사용하되, 각 도구의 `style.css`에서 필요한 만큼 오버라이드 허용
- 다른 도구로 돌아가는 최소한의 네비게이션(예: 좌상단 "← 전체 도구" 링크)만 남기고, 나머지 화면은 해당 도구 전용으로 구성
- 상단에 **통합 내비게이션 바**를 두어 카테고리(계산 도구 | 학습 도구 | 디자인 도구 | 데이터 도구)별로 도구 간 이동을 쉽게 함

### 2-2. 도구 성격에 따른 UI 밀도 구분
| 도구 성향 | UI 방향 |
|---|---|
| 단순 반복 동작 위주 (타이머, 카운터, 국기 다운로드 등) | **핵심 기능만 큰 버튼/큰 숫자로 배치.** 설정 옵션은 접이식(아코디언) 또는 별도 "설정" 화면으로 분리해 첫 화면을 최대한 심플하게 유지 |
| 입력값이 많은 계산 도구 (계단/목공/마이터각도 계산기 등) | 입력 폼은 단계별 그룹핑, 결과는 입력 완료 즉시 하단/우측에 표시. 스크롤 최소화 |
| 파일 업로드/편집형 (이미지 변환, PSD 프리뷰) | 드래그앤드롭 영역을 화면 중심에 크게 배치, 편집 옵션은 사이드 패널로 분리 |
| 전문가용 시공/공학 도구 (마이터각도, 피타고라스, 수축률 계산기) | 현장에서 빠르게 값만 확인하는 용도이므로 입력→결과가 한 화면 안에서 즉시 보이도록, 전문적이고 절제된 톤(다크모드 권장) |

### 2-3. 프롬프트에 포함할 문구 (필요 시 추가)

```
- 이 도구는 [단순 반복 동작 / 복합 입력 / 파일 편집] 성격이니, 그에 맞게 첫 화면 정보량을 조절해 줘.
- 핵심 기능(예: 시작/정지 버튼, 숫자 표시)을 화면에서 가장 눈에 띄게 배치하고, 부가 설정은 접이식 UI로 숨겨줘.
- 다른 도구와 디자인 톤은 다르되, 폰트/버튼 radius 등 기본 스타일 언어는 통일감 있게 유지해 줘.
```

## 4. 개발 프로세스 품질 기준 (DevOps & Quality)

### 4-1. SEO (검색 상위 노출 — 프로젝트 최우선 과제)

#### ① 키워드 전략 (개발 착수 전 반드시 먼저 결정)
- 도구 이름을 정할 때 **사람들이 실제로 검색창에 입력할 법한 단어**를 그대로 반영 (예: "계단 계산기", "무료 국기 다운로드", "팔굽혀펴기 타이머")
- 롱테일 키워드 활용: 경쟁이 낮고 의도가 명확한 검색어 (예: "계단 단높이 계산법", "마이터 각도 계산기")
- 새 도구 기획 시 Claude에게 "이 도구와 관련해 사람들이 구글에 검색할 만한 키워드 10개를 뽑아줘"를 먼저 요청 → 그 키워드를 title/description/본문에 자연스럽게 반영

#### ② 온페이지 SEO (페이지 내 최적화)
- **Title 태그 공식**: `핵심키워드 - 부가설명 | 사이트/브랜드명`
- **Meta Description**: 150자 내외, 키워드 포함 + 클릭을 유도하는 한 줄
- `<h1>`은 페이지당 1개, 핵심 키워드 포함
- 시맨틱 태그 사용: `<div>` 남용 대신 `<header>`, `<main>`, `<article>`, `<section>` 사용
- 이미지에는 항상 의미 있는 `alt` 텍스트 부여

#### ③ 구조화된 데이터 (JSON-LD)
- 각 도구 페이지에 `SoftwareApplication` 스키마 필수 삽입
- 사용법 설명이 있는 경우 `FAQPage` 스키마 추가 → 구글 검색 결과에 FAQ 노출 가능성 ↑

#### ④ 콘텐츠(텍스트) 요건
- 도구 상단/하단에 **"이 도구는 무엇인가", "사용 방법", "이럴 때 유용해요"** 등 최소 300자 이상의 설명 텍스트 배치
- 계산기류는 결과 화면 아래 "계산 방식 설명"을 추가하면 체류시간과 SEO에 모두 도움
- FAQ 3~5개를 텍스트로 노출

#### ⑤ 기술적 SEO
- **sitemap.xml**, **robots.txt** 루트에 생성, 모든 도구 URL 등록
- 의미 있는 URL 경로 사용
- **Core Web Vitals**: LCP 2.5초 이내, CLS 0.1 이하
- **Google Search Console** 등록 + sitemap 제출
- 통합 대시보드 및 도구 간 **내부 링크(관련 도구 추천)** 연결

#### 새 도구 개발 프롬프트에 항상 포함할 SEO 지시문

```
- 이 도구와 관련해 사람들이 실제로 검색할 법한 키워드를 먼저 알려주고, 그 키워드를 title/description/본문에 반영해 줘.
- SoftwareApplication + FAQPage JSON-LD를 함께 작성해 줘.
- 본문에 "도구 소개"와 "사용 방법" 설명을 300자 이상 자연스럽게 포함해 줘.
- sitemap.xml에 이 페이지 URL을 추가해 줘.
```

### 4-2. 성능 최적화
- **Web Worker**: 이미지 변환, PSD 파싱 등 무거운 작업은 Web Worker로 분리해 메인 스레드(UI) 차단 방지
- **Lazy Loading**: 도구별 필요 JS만 로드
- **이미지 최적화**: 캔버스 작업 시 원본 고해상도 대신 리사이징된 미리보기 사용

### 4-3. 보안 및 데이터 보호
- Privacy Policy 문구를 footer 등에 명시: "이 도구는 모든 파일을 브라우저 로컬에서 처리하며, 서버로 전송/저장하지 않습니다"
- HTTPS 필수

### 4-4. 버전 관리
- `feature/도구명` 브랜치에서 개발 → 테스트 후 `main` 병합
- try-catch로 에러 핸들링, 사용자 친화적 메시지 표시

### 추가 지시사항 템플릿

```
- 파일 처리 시 브라우저 응답 없음 방지를 위해 Web Worker를 고려해 줘.
- 데이터가 브라우저 내에서만 처리된다는 점을 명시하는 Privacy 문구를 footer에 포함해 줘.
- 구글 검색 노출을 위해 이 도구 목적에 맞는 SoftwareApplication JSON-LD 스키마를 작성해 줘.
- 파일 변환/파싱 에러를 우아하게 처리하는 에러 메시지 UI를 구현해 줘.
```

## 5. 앱 카테고리별 핵심 로직 가이드

### 5-1. 심볼/리소스 모음 (국기·이모지·특수문자·바코드)
| 카테고리 | 핵심 로직 | 프롬프트 요약 |
|---|---|---|
| 국기 다운로드 | ISO 3166-1 alpha-2 코드 기반 리소스 맵핑. **빌드 시점에 PNG/JPG를 미리 생성해 정적 파일로 저장** (변경 빈도 낮아 런타임 변환은 낭비) | "REST Countries API로 목록을 받아 countries.json으로 저장하고, flag-icons SVG를 받아 Node.js 스크립트(sharp 등)로 사이즈별 PNG/JPG를 미리 변환해 정적 저장. 런타임 변환 없이 `<a download>`로 제공." |
| 이모지 아이콘 모음 (한/영/중/일 검색) | `emojibase-data`(MIT, CLDR 기반)로 ko/en/zh/ja 이름·키워드를 빌드 시 병합. 이미지는 Noto Emoji(Apache 2.0) SVG→빌드 시 PNG 사전 변환. 4개 언어 필드 부분 문자열 검색 | "emojibase-data의 ko/en/zh/ja 데이터를 병합해 emojis.json 생성, Noto Emoji를 PNG로 사전 변환, 다국어 통합 검색 + 다운로드 버튼 구현." |
| 특수문자 모음 (★☆→©® 등) | 유니코드 텍스트 문자 자체라 **저작권 이슈 없음**. 카테고리별 정적 JSON. `navigator.clipboard.writeText()`로 클릭 시 클립보드 복사 | "카테고리별 특수문자 JSON + 카테고리 필터 + 한글 검색. 클릭 시 클립보드 복사 및 토스트 메시지." |
| 바코드 생성기 | **직접 생성형 → 저작권 리스크 없음**. `JsBarcode`(MIT, CDN)로 CODE128/EAN-13/UPC 등 렌더링 후 다운로드. (선택) `ZXing-js`로 웹캠 스캔 검증 | "JsBarcode CDN으로 값+포맷 입력 시 실시간 렌더링, PNG/SVG 다운로드, 잘못된 자릿수는 에러 메시지." |

### 5-2. 생성형 유틸리티 (저작권 리스크 없음)
| 카테고리 | 핵심 로직 | 프롬프트 요약 |
|---|---|---|
| 랜덤 비밀번호 생성기 | `crypto.getRandomValues()`로 암호학적으로 안전한 난수 생성 (`Math.random()` 사용 금지). 길이/대소문자/숫자/특수문자 옵션 체크박스 | "crypto.getRandomValues 기반으로 길이·문자 종류 옵션을 조합한 비밀번호 생성기를 만들어 줘. 강도(약함/보통/강함) 표시기도 추가해 줘." |
| CSS 그라데이션 빌더 | 컬러피커 다중 선택 + 각도/타입(linear/radial) 옵션 → 실시간 CSS 코드 생성 및 클립보드 복사 | "색상 2개 이상 추가/삭제 가능한 컬러피커와 각도 슬라이더로 실시간 미리보기 + CSS 코드 자동 생성, 복사 버튼 구현." |

### 5-3. 학습/집중 도구
| 카테고리 | 핵심 로직 | 프롬프트 요약 |
|---|---|---|

```
   - 타겟 사용자: (누가 검색해서 찾아올지)

2. 핵심 기능 (우선순위 순)
   - 기능 1: ...
   - 기능 2: ...
   - 기능 3(있다면): ...

3. 기술적 접근 방식
   - 핵심 로직: (5번 카테고리 가이드 중 해당 항목 요약)
   - 사용 라이브러리/API: (있다면 이름과 라이선스)
   - UI 성향: [단순 반복형 / 복합 입력형 / 파일 편집형]

4. SEO 방향
   - 예상 검색 키워드: ...

→ 이 방향으로 개발을 시작해도 될까요? 수정하고 싶은 부분이 있으면 말씀해 주세요.
→ "진행해줘"라고 확인해 주시면 바로 코드 작성을 시작하겠습니다.
```

이 확인서는 6-1번 순서표의 해당 도구 정보(핵심 로직/프롬프트 요약)와 5번 카테고리 가이드를 기반으로 Claude Code가 자동으로 채워서 제시합니다.

### 6-3. 프로젝트 완료 시 필수 절차
1. 해당 도구의 핵심 기능 구현 완료
2. Claude Code가 **아래 "점검용 체크리스트"를 자동으로 출력**
3. 사용자가 실제 브라우저에서 수동 테스트 진행
4. 문제 없으면 사용자가 "승인" 또는 "다음 진행" 등으로 명시적으로 확인
5. 확인 후에만 다음 순서의 도구 개발 시작 (커밋/브랜치 병합도 이 시점에 진행) → **6-2번 착수 전 확인 절차부터 다시 반복**

### 6-4. 점검용 체크리스트 템플릿 (매 도구 완료 시 사용)

```
[프로젝트 완료 보고: {도구명}]
✅ 핵심 기능 구현 완료
✅ 모바일/데스크톱 반응형 확인
✅ 에러 핸들링(잘못된 입력/파일) 동작 확인
✅ Privacy 문구 및 광고 슬롯 배치 확인
✅ SEO 메타 태그/JSON-LD 삽입 확인

→ 위 항목을 직접 브라우저에서 테스트해 주세요.
→ 이상 없으면 "다음 진행"이라고 말씀해 주시면 다음 순서({다음 도구명})로 넘어가겠습니다.
```

### 6-5. Claude Code 세션 지시사항 (CLAUDE.md에 포함 권장)

```
- 새 도구 개발에 들어가기 전, 반드시 "개발 착수 확인서"(개요/핵심기능/기술방식/SEO)를 먼저 제시하고 사용자 승인("진행해줘")을 기다릴 것. 승인 전에는 코드 작성을 시작하지 말 것.
- 하나의 도구 개발이 끝나면 바로 다음 도구로 넘어가지 말 것.
- 반드시 "점검용 체크리스트"를 제시하고 사용자의 수동 테스트 및 승인을 기다릴 것.
- 사용자가 "다음 진행" 등으로 명시적으로 확인하기 전까지 다음 프로젝트 코드 작성을 시작하지 말 것.
```

## 7. GitHub Pages + 애드센스 체크리스트

**기술적으로 GitHub Pages(정적 사이트) + 애드센스는 문제없이 가능**하나, 승인 심사를 위한 조건 충족 필요:

- [ ] **커스텀 도메인 연결** — `username.github.io`는 승인이 매우 어려움
- [ ] **개인정보 처리방침 페이지** (`privacy.html`)
- [ ] **콘텐츠 충실도** — 도구별 설명 텍스트 최소 300자 이상
- [ ] **반응형 디자인**
- [ ] 저작권 문제 없는 콘텐츠만 사용

**진행 로드맵**
1. 웹앱 여러 개를 GitHub Pages에 배포
2. 개인 도메인 구매 및 연결
3. `privacy.html` 작성
4. 애드센스 등록 → 광고 코드 삽입 → 승인 신청

**도구 카테고리별 광고 배치 전략 (CTR/CPC 최적화)**
- 계산 도구: 결과값 화면 하단 → 관련 자재/공구 구매 광고 (전환율 높음)
- 학습/타이머 도구: 타이머 종료·휴식 시점 → 강의/교재 광고 (사용자 타겟팅 명확)
- 디자인/변환 도구: 다운로드 버튼 상/하단 → 소프트웨어·디자인 서비스 광고

## 8. Claude Code 협업 팁

1. **"파일 구조를 유지해 줘"** — 기존 도구와 폴더 구조 일관성 유지 요청
2. **"CDN을 사용해 줘"** — npm 설치 불가 환경임을 매 요청 시 재확인
3. **"SEO를 신경 써 줘"** — 메타 태그 최적화 습관화
4. **"다음 도구로 넘어가기 전에 점검 체크리스트부터 줘"** — 6-4번 항목의 완료 후 점검 게이트를 매번 상기
5. **"이 도구 시작하기 전에 착수 확인서부터 줘"** — 6-2번 항목의 착수 전 확인 게이트를 매번 상기
6. 새 세션 시작 시 이 가이드 파일을 CLAUDE.md에 링크하거나 내용을 포함해 컨텍스트 유지
