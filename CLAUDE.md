## 0. 프로젝트 팀 구성 및 언어 정책

### 팀 구성
이 프로젝트는 다음 네 역할로 진행됩니다.
- **PM** — 요청자 겸 최종 의사결정권자
- **기획팀**
- **디자인팀**
- **개발팀** (Claude Code)

### 언어 정책
이 사이트는 **영어(en)가 메인 언어**입니다. bare 도메인으로 접속하면 콘텐츠 없이 `/en/`으로 안내하는 리다이렉트 스텁만 있고, 한국어를 포함한 어떤 언어도 "메인 사이트"처럼 보이지 않도록 모든 언어가 `/{lang}/` 형태로 대칭 구조를 씁니다(자세한 내용은 9-1 참고).

최종적으로 아래 12개 언어를 지원할 계획입니다.

| # | 언어 | 코드 | 상태 |
|---|---|---|---|
| 1 | 영어 (English) | en | ✅ 메인 언어, 구현됨 |
| 2 | 한국어 | ko | ✅ 구현됨 |
| 3 | 일본어 | ja | ✅ 구현됨 |
| 4 | 중국어 간체 | zh | ✅ 구현됨 |
| 5 | 중국어 번체 | zh-TW | ⬜ 대기 |
| 6 | 프랑스어 | fr | ⬜ 대기 |
| 7 | 독일어 | de | ⬜ 대기 |
| 8 | 스페인어(멕시코) | es-MX | ⬜ 대기 |
| 9 | 스페인어 | es | ⬜ 대기 |
| 10 | 포르투갈어 | pt | ⬜ 대기 |
| 11 | 인도네시아어 | id | ⬜ 대기 |
| 12 | 말레이시아어 | ms | ⬜ 대기 |

> ⚠️ **필수 규칙: 새 언어의 번역·개발은 PM이 명시적으로 요청할 때만 진행합니다.** 개발 속도와 효율성을 위한 결정입니다. 이미 구현된 4개 언어(en/ko/ja/zh)에 대해서도 동일합니다 — 새 기능이나 콘텐츠를 만들 때는 **en/ko 위주로 먼저 작업**하고, 나머지 언어(ja/zh 포함)로의 번역은 PM 요청이 있을 때 별도로 진행합니다.
> - 이미 4개 언어로 배포된 페이지의 구조(템플릿 토큰 등)를 바꿔서 그대로 두면 빌드가 깨지는 경우에는 예외입니다 — 이때는 아직 번역되지 않은 언어에 새 번역 콘텐츠를 미리 써넣는 대신, 빌드 스크립트에서 빈 문자열로 안전하게 처리(fallback)해서 "undefined" 같은 깨진 텍스트가 노출되지 않게만 합니다. 이건 번역 작업이 아니라 빌드 안정성 조치입니다.
> - PM이 번역을 요청하면, 그 시점까지 en/ko에 쌓인 변경 사항을 모아 한 번에 번역합니다.

## 🎯 프로젝트 핵심 목표

> **"간단한 기능을 제대로 전달하는 도구를 만들고, 검색을 통해 유입된 사용자로부터 구글 애드센스 광고 수익을 얻는다."**

이 목표에 따라 모든 개발 우선순위는 다음 순서를 따릅니다.

1. **기능 신뢰성** — 도구가 딱 한 가지 일을 정확하고 빠르게 해낼 것 (기능이 많은 것보다 확실한 것이 우선)
2. **검색 상위 노출(SEO)** — 사용자가 이 도구를 실제로 검색해서 찾아올 수 있어야 수익이 발생 (아래 SEO 섹션이 사실상 이 프로젝트의 1순위 기술 과제)
3. **광고 수익화** — 검색 유입 사용자가 이탈하지 않고 광고를 자연스럽게 노출받도록 UX 설계
4. **서버 비용 0원** — GitHub Pages 기반 정적 사이트 유지

> ⚠️ Claude Code에 새 도구 개발을 요청할 때, "기능만 만들어 줘"라고 하지 말고 **"검색 상위 노출까지 고려해서"**를 항상 포함하세요. SEO는 나중에 추가하는 게 아니라 처음부터 설계에 포함되어야 효과가 있습니다.

## 1. 프로젝트 구조 (모듈형)

여러 도구를 하나의 리포지토리에서 관리합니다.

```text
/your-repo-name
  ├── index.html           (통합 대시보드: 모든 도구 링크)
  ├── privacy.html         (애드센스 승인 필수: 개인정보 처리방침)
  ├── /vector-tool
  ├── /stair-calc
  ├── /flag-master
  ├── /pt-timer
  ├── /tempo-counter
  ├── /emoji-search
  ├── /special-char
  ├── /barcode-gen
  └── ... (도구별 폴더, 각각 index.html / style.css / script.js 3분리 구조)
```

**규칙**
- 모든 도구는 `index.html / style.css / script.js` 3분리 구조 유지
- 순수 자바스크립트(서버 로직 없음), 필요 시 CDN(Unpkg/JSDelivr)만 사용
- 모바일 반응형 필수
- 각 도구는 **디자인·UX 상 완전히 독립된 하나의 앱**처럼 느껴지게 구성 (아래 2번 참조)

## 2. 앱별 독립 디자인 & UX 원칙

각 도구는 통합 대시보드에 속해 있지만, **사용자가 그 도구만 쓸 때는 그 도구만의 독립적인 서비스처럼 느껴지도록** 디자인합니다.

### 2-1. 공통 원칙
- 도구마다 목적에 맞는 **고유 컬러 테마/톤**을 부여 (예: 계산기류=차분한 블루/그레이 계열, 운동류=에너지틱한 오렌지/레드 계열, 시공/공학 도구=전문적인 다크모드 계열)
- 공통 대시보드에서 쓰는 공용 CSS 변수(색상, 폰트, 여백 단위)는 재사용하되, 각 도구의 `style.css`에서 필요한 만큼 오버라이드 허용
- 다른 도구로 돌아가는 최소한의 네비게이션(예: 좌상단 "← 전체 도구" 링크)만 남기고, 나머지 화면은 해당 도구 전용으로 구성
- 상단에 **통합 내비게이션 바**를 두어 카테고리(계산 도구 | 학습 도구 | 디자인 도구 | 데이터 도구)별로 도구 간 이동을 쉽게 함

### 2-2. 도구 성격에 따른 UI 밀도 구분
| 도구 성향 | UI 방향 |
|---|---|
| 단순 반복 동작 위주 (타이머, 카운터, 국기 다운로드 등) | **핵심 기능만 큰 버튼/큰 숫자로 배치.** 설정 옵션은 접이식(아코디언) 또는 별도 "설정" 화면으로 분리해 첫 화면을 최대한 심플하게 유지 |
| 입력값이 많은 계산 도구 (계단/목공/마이터각도 계산기 등) | 입력 폼은 단계별 그룹핑, 결과는 입력 완료 즉시 하단/우측에 표시. 스크롤 최소화 |
| 파일 업로드/편집형 (이미지 변환, PSD 프리뷰) | 드래그앤드롭 영역을 화면 중심에 크게 배치, 편집 옵션은 사이드 패널로 분리 |
| 전문가용 시공/공학 도구 (마이터각도, 피타고라스, 수축률 계산기) | 현장에서 빠르게 값만 확인하는 용도이므로 입력→결과가 한 화면 안에서 즉시 보이도록, 전문적이고 절제된 톤(다크모드 권장) |

### 2-3. 프롬프트에 포함할 문구 (필요 시 추가)

```
- 이 도구는 [단순 반복 동작 / 복합 입력 / 파일 편집] 성격이니, 그에 맞게 첫 화면 정보량을 조절해 줘.
- 핵심 기능(예: 시작/정지 버튼, 숫자 표시)을 화면에서 가장 눈에 띄게 배치하고, 부가 설정은 접이식 UI로 숨겨줘.
- 다른 도구와 디자인 톤은 다르되, 폰트/버튼 radius 등 기본 스타일 언어는 통일감 있게 유지해 줘.
```

## 4. 개발 프로세스 품질 기준 (DevOps & Quality)

### 4-1. SEO (검색 상위 노출 — 프로젝트 최우선 과제)

#### ① 키워드 전략 (개발 착수 전 반드시 먼저 결정)
- 도구 이름을 정할 때 **사람들이 실제로 검색창에 입력할 법한 단어**를 그대로 반영 (예: "계단 계산기", "무료 국기 다운로드", "팔굽혀펴기 타이머")
- 롱테일 키워드 활용: 경쟁이 낮고 의도가 명확한 검색어 (예: "계단 단높이 계산법", "마이터 각도 계산기")
- 새 도구 기획 시 Claude에게 "이 도구와 관련해 사람들이 구글에 검색할 만한 키워드 10개를 뽑아줘"를 먼저 요청 → 그 키워드를 title/description/본문에 자연스럽게 반영

#### ② 온페이지 SEO (페이지 내 최적화)
- **Title 태그 공식**: `핵심키워드 - 부가설명 | 사이트/브랜드명`
- **Meta Description**: 150자 내외, 키워드 포함 + 클릭을 유도하는 한 줄
- `<h1>`은 페이지당 1개, 핵심 키워드 포함
- 시맨틱 태그 사용: `<div>` 남용 대신 `<header>`, `<main>`, `<article>`, `<section>` 사용
- 이미지에는 항상 의미 있는 `alt` 텍스트 부여

#### ③ 구조화된 데이터 (JSON-LD)
- 각 도구 페이지에 `SoftwareApplication` 스키마 필수 삽입
- 사용법 설명이 있는 경우 `FAQPage` 스키마 추가 → 구글 검색 결과에 FAQ 노출 가능성 ↑

#### ④ 콘텐츠(텍스트) 요건
- 도구 상단/하단에 **"이 도구는 무엇인가", "사용 방법", "이럴 때 유용해요"** 등 최소 300자 이상의 설명 텍스트 배치
- 계산기류는 결과 화면 아래 "계산 방식 설명"을 추가하면 체류시간과 SEO에 모두 도움
- FAQ 3~5개를 텍스트로 노출

#### ⑤ 기술적 SEO
- **sitemap.xml**, **robots.txt** 루트에 생성, 모든 도구 URL 등록
- 의미 있는 URL 경로 사용
- **Core Web Vitals**: LCP 2.5초 이내, CLS 0.1 이하
- **Google Search Console** 등록 + sitemap 제출
- 통합 대시보드 및 도구 간 **내부 링크(관련 도구 추천)** 연결

#### 새 도구 개발 프롬프트에 항상 포함할 SEO 지시문

```
- 이 도구와 관련해 사람들이 실제로 검색할 법한 키워드를 먼저 알려주고, 그 키워드를 title/description/본문에 반영해 줘.
- SoftwareApplication + FAQPage JSON-LD를 함께 작성해 줘.
- 본문에 "도구 소개"와 "사용 방법" 설명을 300자 이상 자연스럽게 포함해 줘.
- sitemap.xml에 이 페이지 URL을 추가해 줘.
```

### 4-2. 성능 최적화
- **Web Worker**: 이미지 변환, PSD 파싱 등 무거운 작업은 Web Worker로 분리해 메인 스레드(UI) 차단 방지
- **Lazy Loading**: 도구별 필요 JS만 로드
- **이미지 최적화**: 캔버스 작업 시 원본 고해상도 대신 리사이징된 미리보기 사용

### 4-3. 보안 및 데이터 보호
- Privacy Policy 문구를 footer 등에 명시: "이 도구는 모든 파일을 브라우저 로컬에서 처리하며, 서버로 전송/저장하지 않습니다"
- HTTPS 필수

### 4-4. 버전 관리
- `feature/도구명` 브랜치에서 개발 → 테스트 후 `main` 병합
- try-catch로 에러 핸들링, 사용자 친화적 메시지 표시

### 추가 지시사항 템플릿

```
- 파일 처리 시 브라우저 응답 없음 방지를 위해 Web Worker를 고려해 줘.
- 데이터가 브라우저 내에서만 처리된다는 점을 명시하는 Privacy 문구를 footer에 포함해 줘.
- 구글 검색 노출을 위해 이 도구 목적에 맞는 SoftwareApplication JSON-LD 스키마를 작성해 줘.
- 파일 변환/파싱 에러를 우아하게 처리하는 에러 메시지 UI를 구현해 줘.
```

## 9. 다국어(i18n) 지원 아키텍처 (Flag Master·Paste Emoji에 적용, 신규 도구도 이 패턴 따를 것)

> 검색 노출은 "사용자의 검색 언어에 맞는 결과가 나와야" 의미가 있습니다. 브라우저 언어를 감지해 화면 텍스트만 바꾸는 방식은 구글이 그 언어 버전을 별도로 색인하지 못해 SEO 효과가 거의 없습니다. 그래서 **언어마다 실제로 분리된 정적 URL**을 갖는 방식으로 구현합니다.

### 9-1. URL 구조
- 모든 페이지(도구 + 루트 대시보드 + privacy/about)가 예외 없이 `/{lang}/...` 형태로 완전히 대칭입니다: `/ko/{tool}/` · `/en/{tool}/` · `/ja/{tool}/` · `/zh/{tool}/`, 대시보드는 `/ko/` · `/en/` · `/ja/` · `/zh/`.
  - 처음엔 한국어가 `/{tool}/`(prefix 없음)에 있다가 `/ko/{tool}/`로 옮겨 도구는 대칭이 됐는데, 대시보드(`index.html`)만 한동안 "/"에 예외로 남아 있었습니다("← 전체 도구" 링크가 늘 돌아오는 자리라 리다이렉트를 피하려던 것). 이후 특정 언어가 "메인 사이트"처럼 보이지 않도록(글로벌 운영 목적) 대시보드도 `/ko/`로 옮겨 완전히 대칭으로 통일했습니다.
  - 이미 배포되어 색인됐을 수 있는 예전 URL(`/{tool}/index.html`, 그리고 bare `/`)은 콘텐츠 대신 리다이렉트 스텁(`<link rel="canonical">` + 상대경로 `<meta http-equiv="refresh">` + `noindex`)으로 남겨둡니다. 도구 예전 URL은 `/ko/{tool}/`로, bare `/`(루트 대시보드 예전 자리)는 **`/en/`**로 안내합니다 — 도메인만 입력해 들어오는 방문자를 위한 기본값이며, 특정 언어를 편애하지 않기 위해 한국어가 아닌 영어를 기본값으로 씁니다.
  - 도구 페이지 back-link href는 모든 언어가 동일한 공식(`../{lang}/`)을 씁니다 — 예전엔 한국어만 대시보드가 "/"에 있어서 `../`로 예외 처리했지만, 이제 전 언어가 `/{lang}/`에 대칭이라 예외가 필요 없습니다.
- 모든 언어 페이지 `<head>`에 서로를 가리키는 `hreflang` 태그를 넣어, 구글이 검색자의 언어에 맞는 URL을 검색결과에 노출하도록 명시합니다(이 언어별 매칭 자체는 그대로 유지). `x-default`(로케일이 매칭되지 않는 방문자용 기본값)만 **영어**로 설정합니다 — 마찬가지로 한 언어를 "메인"처럼 보이지 않게 하기 위함입니다. `sitemap.xml`에도 `xhtml:link rel="alternate" hreflang="..."`로 동일하게 등록하고, bare `/`(리다이렉트 스텁)는 sitemap에서 제외한 채 `/ko/`를 대시보드 항목으로 등록합니다.

### 9-2. 빌드 파이프라인 (content/template 분리, index.html은 직접 수정 금지)
- 각 도구 폴더(그리고 루트)에 3가지 파일이 있습니다:
  - `template.html` — 언어 무관 뼈대(구조)만, `{{H1}}` `{{FAQ_HTML}}` 같은 토큰 포함
  - `content.ko.json` / `content.en.json` / `content.ja.json` / `content.zh.json` — 언어별 실제 텍스트(title, meta description, h1, 본문, FAQ 등)
  - **`index.html`은 빌드 결과물**입니다. `template.html` + `content.{lang}.json`을 합쳐서 `node scripts/build-i18n.mjs`가 자동 생성합니다. **콘텐츠를 고칠 땐 항상 `content.*.json`을 수정하고 빌드 스크립트를 다시 실행하세요.** `index.html`을 직접 고쳐도 다음 빌드 때 덮어써집니다.
- `scripts/build-i18n.mjs`가 루트 대시보드 + `TOOLS` 배열에 등록된 모든 도구 × `LANGS` 배열(`["ko","en","ja","zh"]`)을 순회하며 도구마다 4개, 대시보드 4개씩 `index.html`을 생성하고, 예전 바건경로에는 리다이렉트 스텁을 씁니다.
- **새 언어 추가 시**: 해당 도구(들)의 `content.{새언어코드}.json`을 작성하고, `script.js`의 `UI_STRINGS`에 그 언어 항목을 추가한 뒤, `scripts/build-i18n.mjs`의 `LANGS` 배열에 코드를 추가하고 빌드를 다시 돌리면 됩니다.
- **새 도구에 다국어 적용 시**: 그 도구 폴더에 `template.html` + `content.ko.json`(+필요한 언어)을 만들고, `scripts/build-i18n.mjs`의 `TOOLS` 배열에 폴더명을 추가하면 나머지는 자동 처리됩니다.

### 9-3. 공유 자산은 절대 언어별로 중복시키지 않는다
- `style.css`·`script.js`·데이터 JSON(`countries.json`/`emojis.json`)·이미지 폴더는 도구 폴더(`{tool}/`)에 **한 벌만** 두고 모든 언어 페이지가 공유합니다.
- 언어별 페이지(`/ko/{tool}/`, `/en/{tool}/` 등)는 실제 폴더 깊이가 다르지만(`{lang}/{tool}/index.html`), `<head>`에 `<base href="../../{tool}/" />`(상대경로!)를 넣어서 `style.css`·`script.js`·`fetch("countries.json")` 같은 상대경로 코드를 **한 줄도 고치지 않고 그대로 재사용**합니다.
  - `<base>`는 반드시 **상대경로**로 씁니다. 절대 URL(운영 도메인)로 쓰면 로컬(`python -m http.server`)에서 열었을 때도 실제 배포된 옛 버전을 불러오게 되는 문제가 있었습니다(직접 겪은 버그).
  - **함정**: `<base>`는 `style.css`/`script.js` 같은 자산 경로뿐 아니라 **페이지 안의 모든 상대경로 링크(`<a href="...">` 포함)**에도 똑같이 적용됩니다. 그래서 각 도구 페이지의 "← 전체 도구" 뒤로가기 링크를 그냥 `href="../"`로 두면, `<base>` 기준으로 다시 계산되어 **언어와 상관없이 항상 한국어 루트(`/`)로 가버리는 버그**가 생깁니다(실사용 테스트로 발견). 그래서 `BACK_LINK_HREF`를 언어별로 따로 계산합니다: 한국어는 `../`, 나머지 언어는 `../{lang}/` (`build-i18n.mjs`의 `backLinkHref` 참고). `<base>`를 쓰는 페이지에 새로운 상대경로 링크를 추가할 때는 항상 이 함정을 의심할 것.

### 9-4. "언어 순수성" 원칙 — 한 언어 페이지에는 그 언어만 나와야 한다
- 사용자 피드백: *"영어 페이지에는 오직 영어만, 일어 페이지에는 오직 일어만 나와야 한다. 섞이면 사용자가 이상하게 느낀다."*
- `script.js`는 `document.documentElement.lang`(=`pageLang`)을 읽어서, 화면에 보이는 **고정 UI 문구**(에러 메시지, 결과 개수 접미사, aria-label, 토스트 메시지, 더보기/접기 버튼 등)를 언어별 테이블(`UI_STRINGS`)에서 가져옵니다.
- 국가명/이모지명처럼 **데이터에서 오는 다국어 텍스트**(`uiLang`)는 원래 브라우저 언어를 자동 감지(`detectLang()`)해서 보여주는 기능이 있었는데, 이걸 `/en/`·`/ja/`·`/zh/` 페이지에도 그대로 적용하면 "영어 페이지인데 한국어 브라우저로 접속하면 국가명만 한국어로 보이는" 언어 혼합 버그가 생깁니다. → **한국어(기본) 페이지만 브라우저 언어 자동 감지를 유지하고, 나머지 언어 페이지는 `pageLang`으로 강제 고정**합니다(`uiLang = pageLang !== "ko" ? pageLang : detectLang()`).
- 새 도구를 다국어로 만들 때도 이 원칙을 반드시 지킬 것: 그 도구의 `script.js`에 하드코딩된 한국어 문구가 있다면 전부 `UI_STRINGS` 언어별 테이블로 옮기고, `pageLang` 기준으로 선택하도록 만들 것.
