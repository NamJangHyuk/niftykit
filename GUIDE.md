# 프로젝트 가이드

작업 중 반복적으로 마주친 문제·상황에서 자연스럽게 추출된 규칙만 기록합니다.

## `<base href="...">`가 걸린 페이지에서 상대경로 링크를 추가/수정할 때

→ **`<base>`가 가리키는 실제 위치(사이트 루트 기준 몇 단계 아래인지)를 먼저 정확히 계산한 뒤, 그 기준으로 상대경로를 다시 계산하라. 다른 페이지 종류의 공식을 그대로 복사하지 말 것.**

- 도구 페이지(`/{lang}/{tool}/index.html`)의 `<base>`는 `../../{tool}/`로, 사이트 루트보다 **한 단계 더 아래**({tool}/ 디렉터리)를 가리킨다. 그래서 뒤로가기 링크는 `../{lang}/`(한 단계 위로) 이어야 한다.
- privacy/about 같은 정보 페이지(`/{lang}/{page}/index.html`)의 `<base>`는 `../../`로, **이미 사이트 루트 자체**를 가리킨다. 여기서 도구 페이지와 똑같이 `../{lang}/`을 쓰면 루트보다 한 단계 더 위로 올라가버려 저장소 경로(`/getniftykit/`)가 통째로 빠지고 404가 난다. `{lang}/`(앞에 `../` 없이)만 써야 한다.
- 실제로 이 프로젝트에서 같은 유형의 실수가 두 번 발생했다(도구 페이지의 뒤로가기 링크 버그, 그리고 privacy/about 페이지의 뒤로가기+footer 링크 버그). `<base>`가 있는 페이지에 새 상대경로 링크를 추가할 때마다 "이 `<base>`는 실제로 어디를 가리키는가"를 먼저 계산하고 검증할 것 — 로컬 서버뿐 아니라 실제 배포 도메인(GitHub Pages의 `/getniftykit/` 하위 경로)에서 링크를 클릭해보거나 Node의 `new URL(href, base)`로 최종 URL을 미리 계산해서 확인하는 것을 권장한다.

## 헤더 공통 컴포넌트(`.top-bar`, `.back-link` 등)를 수정할 때

→ **이 프로젝트는 도구별 완전 독립 디자인 원칙(CLAUDE.md 2장)에 따라 `.top-bar`/`.back-link` CSS가 19개 도구의 `style.css` + 공용 루트 `style.css`(privacy/about용)에 각각 물리적으로 복제되어 있다. 헤더 관련 CSS를 고칠 때는 한 파일만 고치면 되는 게 아니라, `grep -rl "^\.top-bar {" --include="style.css" .`로 전체 대상 파일을 뽑아서 전부 동일하게 패치해야 한다.**

- 공용 컴포넌트를 별도 CSS 파일로 분리해서 import하는 구조가 아니므로, 이런 전체 순회 스크립트(Node로 정규식 삽입 등)를 짜서 한 번에 처리하는 편이 안전하고 빠르다.
